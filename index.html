Enterprise Schema

Problems of User Schema

At 1970s, the database design process was focusing on the user view of the data. However, the user schema is not a 'pure' representation of the world.
There are a few reasons:

  • The database designer is restricted by the limited capabilities of the database management system.
  • User schema might contain some features related to data storage representation
  • User schema is usually designed to be efficient for a certain type of data processing operations.

This makes the user schema difficult to understand and change. Therefore, an enterprise schema was introduced as the intermediate stage of the design process.

Advantage of Using Enterprise Schema

The enterprise schema will be translated into different schemata for different database management systems. It can also be translated into several schemata for the same database management system to optimize different types of data processing operations.
There are several advantages of this approach:

  • Easier to understand: The enterprise schema does not have the restrictions of the underlying database management/system;
  • More stable than the user schema: since some types of changes in the user schema may not require any change in the enterprise schema. If the enterprise schema needs to be changed to reflect the changes in the enterprise environment, the changes can be performed easily since efficiency and storage issues are not considered.

Mental Framework

Since the Entity-Relationship (E-R) model have been found useful in modeling the real world. Entity-Relationship(E-R) Model will be used as the metal framework and E-R diagram will be used to represent the enterprise view of data.

Reference: The Entity-Relationship Model: a basis for the enterprise view of data, PPS Chen AFIPS Proceedings 1977
